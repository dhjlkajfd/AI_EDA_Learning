# Verilog / FPGA / RTL 小项目客户沟通话术模板

本文档用于早期接 Verilog / FPGA / RTL 小项目时复制使用。语气原则是：真诚、稳重、不卑微、不装专家、不承诺包过，也不承诺一定修好。可以先帮对方判断问题类型、工作量和是否适合接。

## 1. 第一次自我介绍话术

### 简短版

你好，我是微电子 / 集成电路方向学生，目前主要做基础 Verilog / FPGA / RTL 小项目。

我可以协助基础 Verilog 模块、Testbench、仿真 Debug、README 和设计文档整理。最近完成了 UART TX、UART RX、UART Loopback 三个练习项目，包含 RTL、自检查 Testbench、Icarus Verilog 仿真、一键仿真脚本和文档。

如果你的需求是基础 RTL 小模块或仿真问题，可以先把需求和报错发我，我可以先判断问题类型和大概工作量。

### 稍正式版

你好，我目前是微电子 / 集成电路方向学生，正在做基础 Verilog / FPGA / RTL 项目积累。

我能做的范围主要是：

- 基础 Verilog 小模块；
- Testbench / 自检查 Testbench；
- Icarus Verilog、Vivado、ModelSim 仿真问题排查；
- README、设计文档、仿真说明整理；
- 初学者 RTL 学习辅助。

我不接明显代写和超出能力的大型项目，也不会承诺包过。可以先看你的需求，判断是否适合接，以及大概需要多少时间。

## 2. QQ 群看到别人问 Verilog 报错时的回复

### 群内公开回复

这个报错看起来可能和端口连接 / 位宽 / 语法格式有关。你可以把完整报错、相关模块代码和仿真命令贴一下，只看一行报错不太好判断。

如果方便，也可以私发我，我可以先帮你判断是编译问题、Testbench 问题，还是 RTL 逻辑问题。

### 更主动但不冒犯的回复

这个问题建议先看三点：

- 报错对应的是哪一行代码；
- 模块端口名和例化端口是否一致；
- 信号位宽是否匹配。

你把完整报错和相关代码发出来，我可以帮你先判断一下方向。

## 3. 对方说“能不能帮我看看代码”的回复

可以。你先把以下内容发我：

- 代码文件或关键代码片段；
- 你现在遇到的问题；
- 编译 / 仿真的完整报错；
- 你期望的结果；
- 当前实际结果；
- 使用的工具，比如 Icarus Verilog、Vivado 或 ModelSim。

我可以先帮你判断问题类型和大概工作量。如果只是简单问题，我可以直接指出；如果需要深入改代码或写 Testbench，我们再确认价格和交付范围。

## 4. 对方问价格时的回复

### 通用回复

价格要看具体工作量。我一般会先看需求，再判断是简单 Debug、Testbench 编写，还是完整小模块开发。

参考范围大概是：

- 简单仿真 / 报错 Debug：50 - 300 元；
- Testbench 编写：100 - 600 元；
- 基础 Verilog 小模块开发：100 - 500 元；
- README / 设计文档整理：50 - 300 元。

你可以先把需求和现有代码发我，我先判断工作量，再给你一个明确报价。

### 对方只问“多少钱”时

需要先看需求才能准确报价。不同任务差别比较大，比如只是修一个端口错误，和从零写 RTL + Testbench + 文档，工作量不一样。

你先发一下需求、代码、报错和截止时间，我看完后给你一个明确价格和交付时间。

## 5. 对方需求不清楚时的追问清单

你这个需求我需要再确认一下，避免理解错。麻烦补充这些信息：

1. 要实现什么功能？
2. 输入信号有哪些？
3. 输出信号有哪些？
4. 时钟频率是多少？
5. 复位是高有效还是低有效？
6. 有没有指定模块名和端口名？
7. 需要写 RTL，还是只需要改已有代码？
8. 是否需要 Testbench？
9. 是否需要自检查 Testbench？
10. 使用什么仿真工具？
11. 现在有没有报错？如果有，请发完整报错。
12. 截止时间是什么时候？
13. 是否需要讲解设计思路？
14. 最终交付形式是代码、文档，还是两者都要？

信息越完整，我越能准确判断工作量和报价。

## 6. 接单前确认交付范围的话术

我先确认一下这次的交付范围，避免后面理解不一致。

本次我负责：

- 编写 / 修改 Verilog RTL；
- 编写 / 修改 Testbench；
- 保证仿真可以跑通；
- 提供基本 PASS / FAIL 检查；
- 简单说明修改点和运行方法。

本次不包含：

- 上板验证；
- Vivado 工程完整搭建；
- 复杂约束文件；
- 大型系统重构；
- 超出当前需求的新功能。

如果你确认这个范围，我再开始做。

## 7. 确认价格和时间的话术

我看了一下你的需求，工作量主要在：

- 理解现有代码；
- 定位仿真 / 逻辑问题；
- 修改 RTL 或 Testbench；
- 重新跑仿真；
- 整理修改说明。

这次报价是：XX 元。

预计交付时间：XX 小时 / XX 天内。

交付内容包括：

- 修改后的代码；
- 仿真结果说明；
- 关键修改点说明；
- 如有需要，补充简单运行步骤。

我不能承诺一定包过或适配所有隐藏要求，但会按当前你提供的需求和代码完成可复现的修改。如果中途发现需求和原描述差异很大，需要重新确认范围和价格。

## 8. 交付时的话术

我这边已经完成本次修改，交付内容如下：

- 修改文件：XXX；
- 新增文件：XXX；
- 仿真工具：XXX；
- 仿真结果：PASS / 已复现问题 / 已定位到原因；
- 主要修改点：XXX。

建议你先按我提供的步骤在本地跑一遍。如果运行环境不同，可能会有路径或工具版本差异，可以把新的报错发我，我再帮你判断。

## 9. 交付后请求反馈的话术

你本地跑完后，如果结果正常，麻烦给我一个反馈。

如果方便的话，也可以简单评价一下这次交付：

- 是否解决了你的问题；
- 说明是否清楚；
- 后续还有没有类似 Verilog / Testbench / 仿真 Debug 的需求。

这对我后续优化接单流程和作品集整理有帮助。

## 10. 礼貌拒绝超出能力范围项目的话术

这个需求我看了一下，已经超出我当前能稳定交付的范围了。

我目前更适合做基础 Verilog 小模块、Testbench、仿真 Debug 和文档整理。你这个任务涉及的内容比较复杂，例如大型系统集成、AXI、CPU、复杂 SoC、上板联调或完整工程交付，我不想为了接单而乱承诺。

建议你找有相关项目经验的人来做。这个我暂时不接，避免耽误你的时间。

## 11. 礼貌拒绝明显代写 / 作弊需求的话术

这个需求如果是考试、考核或明确要求代写后伪装成你自己完成，我这边不能接。

我可以做的是：

- 帮你解释题目；
- 帮你梳理设计思路；
- 帮你检查你自己写的代码；
- 帮你分析报错；
- 帮你理解 Testbench 和波形。

但不能直接替你完成明显属于考核性质的内容，也不能帮你伪造过程或结果。

## 12. 对方只白嫖、不愿意说明需求时的边界话术

只发一句“帮我看看”或者只截一小段图，我很难判断问题，也容易浪费双方时间。

如果你希望我认真帮你看，请至少提供：

- 完整报错；
- 相关代码；
- 运行命令；
- 期望结果；
- 实际结果。

我可以先免费判断问题大方向。如果需要深入排查、修改代码或写 Testbench，就需要按任务报价。

## 13. 对方催得很急时的回复

我理解你比较急，但 Verilog / 仿真问题需要先复现和定位，不能只靠猜。

如果你希望我接，需要先把代码、报错、工具版本和截止时间发完整。我会判断能不能在你的时间内完成。

如果时间太紧，而问题又比较复杂，我会直接说明不接，避免最后交付不了。

## 14. 对方预算很低时的回复

这个预算我理解，但按你的需求来看，需要阅读代码、定位问题、修改并重新仿真，工作量已经超过这个价格了。

如果预算有限，可以缩小范围，比如：

- 我只帮你定位问题，不改完整代码；
- 我只给修改建议，不写完整 Testbench；
- 我只整理运行步骤，不做文档；
- 先做最关键的一个 bug。

如果你愿意缩小范围，我可以重新按小任务报价。

## 15. 常用短句

### 判断工作量

我先看一下完整需求和代码，再判断能不能接，以及大概多少钱。

### 不承诺包过

我可以按你提供的需求完成代码和仿真，但不承诺包过或满足未说明的隐藏要求。

### 需要完整信息

目前信息不够，不能准确判断问题。请先补充完整报错、代码和运行环境。

### 先小范围确认

建议先把任务拆小，先确认 RTL 或 Testbench 是否能跑通，再继续加功能。

### 收到文件

文件我收到了。我先看结构和报错，之后给你反馈问题类型、预计时间和报价。

### 修改范围变化

这个已经超出最初确认的范围了。如果需要继续做，我们需要重新确认需求、时间和价格。

## 16. 推荐沟通流程

1. 先了解需求。
2. 要完整代码和报错。
3. 判断是否在能力范围内。
4. 确认交付内容。
5. 确认价格和时间。
6. 开始修改或编写。
7. 本地仿真验证。
8. 交付代码、结果和说明。
9. 收集反馈。
