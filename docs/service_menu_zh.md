### B. Testbench 编写

可做内容：

- 普通 Testbench。
- 自检查 Testbench。
- 带 timeout 的 Testbench。
- 基础输入激励生成。
- 输出结果自动检查。

早期参考价：

可交付内容：

## 5. 客户需要提供什么

为了提高沟通效率，请尽量提供以下信息：

| 信息 | 说明 |
|---|---|
| 需求说明 | 想实现什么功能，输入输出是什么，有没有时序要求 |
| 原始代码 | RTL、Testbench、约束文件或已有工程文件 |
| 报错截图 | 编译报错、仿真报错、工具输出、终端日志 |
| 仿真工具版本 | Icarus Verilog、Vivado、ModelSim 等工具及版本 |
| 期望交付形式 | 只要修改建议、需要代码、需要文档、还是需要讲解 |
| 截止时间 | 是否紧急，是否分阶段交付 |
| 是否需要讲解 | 只交付结果，还是需要解释设计思路和修改原因 |

如果是 Debug 类任务，最好同时提供：

- 完整工程目录或最小可复现代码。
- 当前运行命令。
- 期望现象。
- 实际现象。
- 关键波形截图。

## 6. 交付方式

根据任务类型，交付内容可以包括：

- `.v` Verilog RTL 文件。
- `.v` Testbench 文件。
- PowerShell / Tcl / 命令行仿真脚本。
- Markdown README 或设计文档。
- 仿真 PASS / FAIL 结果说明。
- 修改点说明。
- 学习讲解笔记。

默认优先交付清晰、可复现、容易继续学习和维护的结果，而不是只给一段无法解释的代码。

## 7. 报价说明

以上价格是早期参考价，实际报价会根据以下因素调整：

- 模块复杂度。
- 是否需要自检查 Testbench。
- 是否需要完整文档。
- 是否需要远程讲解。
- 是否需要快速交付。

简单问题可以先低价沟通；复杂需求建议先拆成小阶段，例如先做需求确认，再做 RTL，再做 Testbench，再做文档整理。
